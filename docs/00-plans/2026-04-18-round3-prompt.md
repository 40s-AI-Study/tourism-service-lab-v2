# [Round 3] 기존 아이디어 고도화·융합 — 사용자·감정 관점 렌즈 적용

> 이 문서는 **Paperclip 7 에이전트**가 Round 3 작업을 수행할 때 참조하는 **설계 프롬프트**입니다.
> TOU 이슈 본문 + 에이전트 지침 참조용으로 함께 사용됩니다.

## ⚠️ 이 이슈의 핵심 의도 (반드시 먼저 읽기)

Round 1·2에서 확보한 **20개 아이디어 + 200건 시뮬 + 27개 API 카탈로그**는 소중한 자산입니다.
**싹 갈아엎지 마세요.**

다만 지금까지 평가·기획 관점이 **API 활용·시장 규모 중심**이었고, **감정·UI/UX·사용자 관점**은 약했습니다.
그 결과 프로토타입이 트리플·야놀자·Klook·네이버지도와 감정적으로 구분이 안 됩니다.

**Round 3 목표**:
1. 기존 자산 위에 **새 관점 6축**을 덧씌워서
2. 기존 아이디어를 **고도화·융합**하거나
3. 필요한 경우에만 **신규 아이디어** 추가

**"기존 vs 신규"가 아니라 "기존 × 신규 관점 = 더 양질의 아이디어"** 입니다.

---

## 🧭 새 관점 6축 (기존 평가에 추가하는 렌즈)

기존 5축(시장 규모·경쟁 강도·API 활용도·구현 난이도·공모전 적합성)은 **그대로 유지**. 여기에 다음 6축 추가:

| # | 축 | 질문 | 측정 방법 |
|---|---|---|---|
| 1 | 감정 JTBD | 왜 이 앱을 여는가? 일상탈출·힐링·스트레스 해소 중 무엇? | 감정 한 줄 명시 가능한가 |
| 2 | 성향 다양성 | 5성향(조용/활달/미식/전통감성/계절사진) 중 몇 개 커버? 섞인 mood 수용? | 성향별 시나리오 3개 작성 가능한가 |
| 3 | 숨은 명소 발굴력 | 서울·동해·서해·리조트 쏠림 완화? 비-상위-50 POI 자연 노출? | 추천 메커니즘 서술 가능한가 |
| 4 | 4계절 콘텐츠화 | 같은 장소의 봄/여름/가을/겨울이 다른 콘텐츠? | 계절별 분기 존재하는가 |
| 5 | 국내/외국인 이중 렌즈 | 동일 POI가 두 집단에 다른 의미? | 한국인 view / 외국인 view 각 1문장 서술 가능한가 |
| 6 | UI/UX 감정 차별화 | 첫 화면이 감정을 묻는가? 트리플·야놀자와 구별? | Hero Screen 서술 가능한가 |

**통합 평가 방식**: 기존 5축(총 100점) + 신규 6축(각 1-10점, 총 60점) = **총 160점**
- 신규 6축 합계 **≥35/60** 이어야 고도화 후보 진출
- 통합 160점 기준 상위 3개를 최종 선정

## 📋 프로세스 (에이전트별 업무)

### Step 1 — 🔍 리서치님 (MR): 추가 리서치 (기존 위에 쌓기)

기존 `knowledge-base/market-research/` 파일은 유지. **부족한 부분만 보강**:

1. **감정·힐링 기반 여행 서비스 글로벌 벤치마크**
   - Calm (명상), Headspace (수면)
   - Airbnb Experiences (체험 큐레이션)
   - Culture Trip (감성 가이드)
   - Strava (운동 감정), Spotify (mood 플레이리스트 UX)
   - 일본 "Roadmovie", 대만 "KKday Experiences"
2. **성향별 여행 소비 트렌드** (조용/활달/미식/전통/사진) 정량 데이터
3. **숨은 명소 신규 데이터 소스** 발굴: 지자체 관광과 자료, KTO 보도자료, 지역 블로그 빅데이터 크롤링 가능성
4. **한국인 여행자의 "일상탈출" 동기** 설문·조사 근거

산출물: `knowledge-base/market-research/round3-추가-리서치.md` (기존 파일 덮어쓰지 말 것)

### Step 2 — 💰 분석님 (BA): 기존 아이디어 20개 6축 재평가

기존 20개 아이디어 전부에 새 6축 점수 매기기. 기존 5축 점수는 그대로 유지.

산출물: `knowledge-base/business/round3-6축-재평가.md`
- 표: 아이디어 × 6축 × 근거(원문 인용 필수)
- 총 160점 기준 재순위
- **기존 PASS 12개 중 6축에서도 강한 것** vs **기존 FAIL이었지만 6축으로 보면 숨은 진주** 분류

### Step 3 — 📋 기획님 (PM): 3갈래 작업

BA 재평가 결과를 토대로 세 갈래:

#### 갈래 A — 기존 아이디어 고도화 (우선)
6축 고득점 기존 아이디어 3-5개를 골라, 약한 축을 보강한 **v2 문서** 작성.
- 저장: `knowledge-base/ideas/{기존이름} v2.md` (원본 유지, v2 병존)
- 변경 지점 명시: "이 아이디어의 감정 JTBD를 ___로 재정의, Hero Screen을 ___로 재설계"

#### 갈래 B — 기존 아이디어 융합
예: "Audio Story Korea + SilverTrail Korea + 전통 감성" = 새 통합 아이디어
- 저장: `knowledge-base/ideas/{새이름} (융합).md`
- 모 아이디어 링크 + 융합 근거 (왜 따로보다 합친 게 나은가)

#### 갈래 C — 신규 발상 (필요 시만)
6축 중 기존 20개로 커버 못 하는 축이 있을 때만.
- 저장: `knowledge-base/ideas/{신규이름}.md`
- **왜 기존으로 안 되는지** 명시 (유사도 검증 필수)

**목표**: 갈래 A 3-5개 + 갈래 B 1-2개 + 갈래 C 0-2개 = 총 5-10개 "업그레이드 아이디어"

각 문서 필수 포함:
- **첫 화면(Hero Screen)**: 말로 묘사
- **감정 JTBD 한 줄**: "사용자가 ___할 때 ___하려고 앱을 연다"
- **기존 대비 차별점**: 트리플/야놀자/Klook **+ 기존 Round 1·2 아이디어**와 어떻게 다른가
- **성향별 시나리오 3개**
- **계절별 분기 + 국내/외국인 렌즈**
- **공모전 심사 기준 매핑** (아래 심사 기준 섹션 참고)
- **활용 API**: 한글 파일명 기준

### Step 4 — 🎨 디자인님 (UX): Hero Screen 와이어프레임
각 업그레이드 아이디어에 **Hero Screen + 핵심 플로우 1개** 문서화. 감정 중심.

### Step 5 — 🎭 시뮬님 (Sim): 페르소나 시뮬
업그레이드 아이디어 × 10 페르소나. 
평가 관점 전환: Pass/Fail + **"이 페르소나가 앱을 다시 열까? (감정·리텐션)"**

### Step 6 — 🎯 전략님 (CEO): 최종 선정
Sim + 160점 통합 점수 → **최종 상위 3개**. 팀 피어 리뷰 + 합의 리뷰 파일.

### Step 7 — 🎨 UX: 프로토타입 HTML (아래 상세 요구사항)

상위 3개 프로토타입 HTML 제작.
`docs/prototypes/round3-{slug}/index.html`.
`docs/prototypes/index.html`에 **Round 3 섹션** 추가.

## 🎨 프로토타입 HTML 필수 포함 섹션 (사용자 요구)

이전 Round 1·2 프로토타입은 UI 중심이라 **서비스 배경·근거·평가가 누락**되었습니다.
Round 3 프로토타입은 **한 페이지에서 서비스의 전체 맥락을 이해할 수 있어야 합니다.**

각 프로토타입 HTML에는 아래 8 섹션 모두 포함:

### §1. Hero (첫 인상)
- 서비스 이름 + 이모지·로고
- 감정 JTBD 한 줄 (커다랗게)
- 첫 화면 모의 UI 1개

### §2. 서비스 개요 (What)
- 무엇을 만드는가 — 200자 이내
- 어떤 사람을 위한 서비스인가 (성향 스펙트럼)
- 기존 트리플·야놀자·Klook과 다른 점

### §3. 리서치 근거 (Why — 왜 이걸 만드나)
- 이 서비스가 필요한 근거가 된 리서치 출처 3-5개
  * `knowledge-base/market-research/` 파일명 + 인용 문장
  * 글로벌 벤치마크 1-2개
  * 한국 관광 문제 통계 1-2개
- 각 근거마다 **한 줄 요약 + 원문 링크**

### §4. 아이디어 도출 과정 (How — 어떻게 이 아이디어까지 왔나)
- 갈래 분류: A 고도화 / B 융합 / C 신규 중 무엇?
- 모 아이디어(갈래 A·B의 경우) 링크
- 새 관점 6축 중 어느 축에서 강해져서 선정됐는가
- BA 재평가 점수(160점 중 몇 점)

### §5. 공모전 평가 기준 대응 (How does it win)
공모전 심사 기준 100점 매핑:

| 심사 항목 | 배점 | 이 서비스의 대응 |
|---|---|---|
| 서비스 기획력 (구체성·독창성·트렌드) | 30 | (구체 설명) |
| 서비스 완성도 (기능성·안정성·편의성) | 30 | (구체 설명) |
| 데이터 활용 적절성 (공사 OpenAPI 필수) | 20 | 한글 파일명 기준 활용 API 나열 |
| 서비스 발전성 (지속성·확장성) | 20 | (구체 설명) |

+ 가점 항목: Start-up NEST (+2), 지역 특화 (+2)

### §6. 주요 화면 (Mock UI)
- Hero Screen 1개
- 주요 기능 화면 3-5개 (와이어프레임·모의 UI)
- 사용자 여정 (예: 앱 설치 → 첫 화면 → 감정 선택 → 추천 → 방문 → 공유)

### §7. OpenAPI 활용 (한글 파일명 기준)
각 활용 API에 대해:

| 한글 API 파일명 | 서비스 내 역할 | 호출 빈도·캐시 |
|---|---|---|
| `한국관광공사_관광지 집중률 방문자 추이 예측 정보` | 숨은 명소 혼잡도 표시 | 1시간 TTL |
| ... | ... | ... |

활용 API 수는 자랑이 아니라, **실제로 UX에 기여하는 API만** 나열 (4-6개 권장).

### §8. 팀 합의 점수 + 근거 링크
