# SIM-185: KoreaDemand Atlas × David Park (외국인 30대 블레저)

## 시나리오

서울 출장 중인 David는 Reddit r/koreatravel에서 KoreaDemand Atlas를 소개하는 스레드를 발견한다. "데이터 기반으로 덜 알려진 여행지 찾는 앱"이라는 설명이 PM 관점에서 흥미롭다. 주말 이틀을 자유 시간으로 확보했고, 경주 당일치기를 생각하던 차에 앱을 설치한다. 영어 UI로 전환하면 경주 주변 수요 데이터와 방문자 집중 예측이 보인다. UX에 높은 기준을 가진 그는 첫 3분 안에 앱이 유용한지 판단한다.

---

## 사용자 여정

### 1단계: 진입
- Reddit 스레드 링크 → 앱스토어 설치 → 영어 전환 즉시 완료
- 첫 화면 히트맵: 전국 수요 강도 직관적 시각화 → "이거 꽤 잘 만들었는데" 즉각 반응
- PM 관점에서 API 데이터 소스(신용카드·이동통신)가 명시된 점에 신뢰감 형성

### 2단계: 행동
- 경주 검색 → 방문자 집중률 예측: 토요일 오전 집중, 일요일 오후 상대적 여유 확인
- 경주 관광 잠재력 점수 71점, 관광 다양성 높음(문화·자연·음식 고루 발달) 확인
- 핵심 명소 영어 정보: 불국사·석굴암·대릉원 등 상세 설명 → 역사 배경 파악 가능
- "반나절 코스" 필터 원하나 없음 → 전체 코스가 하루 기준으로만 생성되어 시간 단위 커스터마이징 불가
- 데이터 지표의 의미를 정확히 알고 싶어 "이 점수가 어떻게 산출됨?" 탐색했으나 설명 없음 → PM으로서 불투명성 거슬림

### 3단계: 결과
- 방문 타이밍 데이터를 활용해 일요일 오후 경주 방문 → 예측과 일치, 비교적 한산
- 핵심 명소 영어 정보는 TripAdvisor 수준으로 유용
- 전반적 평가: "Google Maps보다 낫진 않지만, 혼잡도 예측 기능은 Google에 없는 차별점"

## 평가

| 항목 | 점수 (1-10) | 근거 |
|---|---|---|
| 유용성 | **7** | 방문 타이밍 예측과 영어 지역 정보 조합이 블레저 여행자의 제한된 시간 최적화에 실질 도움 |
| 사용 편의 | **6** | 영어 UI 자체는 양호하나 "시간 기반 코스 추천" 부재와 점수 산출 방식 불투명성이 UX 품질에 민감한 David를 불편하게 만듦 |
| 구현 타당성 | **8** | 영어 지원 + 예측 API 조합은 충분히 구현 가능. 해외 사용자 대상 PMF 확보 가능성 있음 |

**종합 점수: 21/30**

## Pass/Fail 판정

> **✅ PASS**

KoreaDemand Atlas는 David Park의 "한국에서 Google Maps 대체제" 니즈를 부분적으로 충족하며, 특히 방문자 집중 예측이라는 Google에 없는 기능으로 차별적 가치를 전달한다. 유용성 7점, 종합 21점으로 기준 충족. 단, UX 품질에 예민한 PM 사용자를 완전히 만족시키려면 데이터 투명성과 시간 단위 코스 기능이 필요하다.
