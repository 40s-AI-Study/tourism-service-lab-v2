# SIM-127: K-BeautyMed Tour × 이재호 (전동휠체어 개발자)

## 시나리오

재호는 개발자 커뮤니티 슬랙에서 K-BeautyMed Tour API 구성에 대한 기술 토론을 보고 호기심에 앱을 설치했다. `medical-tourism` API와 `visitor-concentration-forecast` 조합에 대한 기술적 관심이었다. 앱을 열자마자 INTJ 성향대로 기능 구조를 빠르게 파악했다. 그러나 전동휠체어 사용자로서 가장 필요한 정보 — 피부과 클리닉의 휠체어 진입 가능 여부, 경사로 상태, 전동휠체어 충전소 — 는 앱 어디에도 없었다. 기술적 구현은 흥미롭지만 자신이 실제로 쓸 앱이 아님을 2분 만에 결론 내렸다.

---

## 사용자 여정

### 1단계: 진입
- 개발자 커뮤니티 기술 토론에서 API 활용 사례로 앱 인지 → 기술적 호기심으로 설치
- 앱 아키텍처와 API 활용 방식 평가 목적. 본인 여행 용도가 아님

### 2단계: 행동
- 기능 탐색: 피부과 목록, 시술 후 쇼핑 코스, 포토스팟 큐레이션 — 구현 방식은 합리적으로 평가
- 병원 상세 페이지 확인: 무장애 접근성 정보(휠체어 진입 가능 여부, 엘리베이터, 주차) 완전 부재
- `barrier-free-travel` API 미활용 — 재호 입장에서 가장 중요한 데이터가 빠진 설계
- "의료 앱인데 장애인 접근성 데이터가 없다는 게 아이러니하네" 내부 평가

### 3단계: 결과
- 기술 리뷰 목적 달성 후 앱 삭제. 여행 사용 전환 없음. 개발자 커뮤니티에 "장애인 접근성 데이터 누락" 피드백 코멘트 게시

## 평가

| 항목 | 점수 (1-10) | 근거 |
|---|---|---|
| 유용성 | **2** | 전동휠체어 사용자에게 의료 클리닉 무장애 정보 없음. 핵심 니즈 미충족 |
| 사용 편의 | **6** | UX 구조는 개발자 시각에서 합리적. 장애인 접근 경로 자체가 설계되지 않음 |
| 구현 타당성 | **8** | 기술 구현 자체는 타당. `barrier-free-travel` API 추가로 개선 여지 있음 |

**종합 점수: 16/30**

---

## Pass/Fail 판정

> **⚠️ CONDITIONAL PASS**

기술 구현 타당성은 높으나 전동휠체어 사용자라는 핵심 페르소나 니즈(무장애 접근성)가 완전히 누락되어 실용 전환이 불가능하다. 의료 시설을 다루는 앱으로서 장애인 접근성 데이터 부재는 구조적 결함이다. `barrier-free-travel` API를 병원 상세 정보에 통합하면 이 페르소나에게도 유의미한 서비스가 될 수 있다.

---

## 개선 제안

1. 병원·클리닉 상세 페이지에 `barrier-free-travel` API 데이터 통합 — 휠체어 진입 가능 여부, 주차, 엘리베이터 정보 필수 항목화
2. 의료관광 앱의 특성상 시술 후 이동이 불편한 환자를 위한 접근성 정보는 모든 사용자에게 유용 — 장애인 전용이 아닌 "편의시설 상세" 탭으로 범용화
