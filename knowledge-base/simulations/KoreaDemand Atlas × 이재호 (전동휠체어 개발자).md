# SIM-187: KoreaDemand Atlas × 이재호 (전동휠체어 개발자)

## 시나리오

재호는 GitHub에서 한국관광공사 API 활용 프로젝트를 둘러보다 KoreaDemand Atlas를 발견한다. 개발자로서 API 구성에 먼저 관심이 생기지만, 곧 "방문자 집중률 예측"이 전동휠체어 사용자에게 실질적으로 유용한 기능임을 인식한다. 붐비는 관광지는 휠체어 이동 공간이 좁아지는 문제가 있어, 한산한 시간대 방문은 안전 문제와 직결된다. 앱을 설치해 부산 여행에 활용해보기로 한다.

---

## 사용자 여정

### 1단계: 진입
- GitHub 연관 프로젝트 링크 → 앱스토어 설치 (개발자 특유의 기술적 경로)
- 첫 화면 히트맵에서 API 데이터 소스(이동통신·신용카드·빅데이터) 명시 확인 → 신뢰도 높게 평가
- 부산 선택 → 지역별 수요 강도·방문자 집중률 데이터 탐색

### 2단계: 행동
- 부산 해운대 vs. 감천문화마을 비교 → 해운대 주말 집중률 92%, 감천 58% — 감천 선택
- 감천문화마을 명소 정보 클릭: 경사로, 계단 정보 없음 → 가장 필요한 정보 부재 확인
- 관광 다양성 지수·잠재력 점수는 휠체어 접근성과 무관한 지표임을 인식 → 핵심 니즈 미충족
- 방문자 집중률 낮은 시간대(화요일 오전) 확인은 가능 → 이동 공간 확보 용도로 활용 결정
- 개발자 시각: "API에 barrier-free 데이터가 없으니 당연히 이 앱에도 없겠지"라고 이해

### 3단계: 결과
- 화요일 오전 감천 방문 → 실제 한산, 전동휠체어 이동 비교적 수월
- 그러나 현장에서 일부 골목 계단 발견 — 사전에 알 수 없었던 정보
- "혼잡도 예측은 쓸 만한데, 내가 진짜 필요한 건 이 앱이 해결 못 해줌"이라는 냉정한 평가

---

## 평가

| 항목 | 점수 (1-10) | 근거 |
|---|---|---|
| 유용성 | **5** | 혼잡도 예측이 이동 공간 확보 측면에서 간접적 유용성 제공. 그러나 접근성 핵심 니즈(경사도·단차·무장애 경로)는 완전히 미충족 |
| 사용 편의 | **8** | 개발자 배경의 재호에게 데이터 인터페이스는 직관적. 앱 구조 파악이 빠름 |
| 구현 타당성 | **7** | 현재 API 범위에서는 무장애 데이터 통합 불가 — 별도 API 연동 필요 |

**종합 점수: 20/30**

---

## Pass/Fail 판정

> **✅ PASS**

종합 점수 20점, 유용성 5점으로 기준을 충족한다. 재호의 데이터 리터러시가 앱 활용을 가능하게 하며, 혼잡도 예측이 휠체어 이동 공간 확보라는 간접적 접근성 가치를 제공한다. 그러나 이 PASS는 재호의 기술 친숙도가 낮은 장애인 사용자에게는 적용되지 않는다. 무장애 정보 미지원이 앱의 명확한 한계다.

---

## 개선 제안

1. `barrier-free-travel` API 연동 레이어 추가 — 현재 빅데이터 지수 4종에 무장애 데이터 5번째 레이어로 통합해 "전동휠체어 이동 가능 지역" 필터 제공
2. 방문자 집중률 예측에 "혼잡 시 좁은 통로 주의" 경고 태그 추가 — 이동약자 사용자에게 혼잡도 데이터의 의미를 명시적으로 연결
