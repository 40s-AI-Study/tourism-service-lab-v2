# SIM-197: TasteData Korea × 이재호 (전동휠체어 개발자)

## 시나리오

재호는 파트너와 함께 전주 주말 여행을 준비하면서 TasteData Korea를 발견한다. 평소 음식을 즐기고 "진짜 현지 맛집"에 관심이 높은 재호에게 "신용카드 데이터 기반 현지인 맛집"이라는 가치는 매력적이다. 개발자로서 앱 구조를 빠르게 파악하고, 전주 비빔밥·한정식·막걸리 소비 강도 지역 데이터를 탐색한다. 그러나 본인이 전동휠체어를 사용한다는 현실에서 핵심 질문이 생긴다. "이 맛집 지역이 전동휠체어로 접근 가능한가?"

---

## 사용자 여정

### 1단계: 진입
- 개발자 커뮤니티 채널에서 앱 소개 글 발견 → 설치
- 전주 검색 → 현지인 소비 히트맵 + 지역 특산 음식 데이터 탐색 (API 구조에 관심)
- 비빔밥·한정식 소비 강도 지역(객사길·전동) 확인 → 방문 대상 지역 선정

### 2단계: 행동
- 객사길 지역 탭: 음식 소비 강도 높음, 지역 특산 음식 정보 풍부 — 맛집 발굴 목적으로 만족
- 휠체어 접근성 필터 탐색 → 없음 (예상했던 결과)
- 개발자 시각: "tourism-big-data API에 접근성 데이터 포함되어 있을 가능성 낮다는 건 알지만, 없으니 별수 없지"
- 음식 코스 생성: 점심 비빔밥 한정식 → 막걸리 전통주 → 콩나물국밥 코스 저장
- 생성된 코스의 식당들이 전동휠체어 접근 가능한지 → 결국 개별 전화 확인 필요

### 3단계: 결과
- 객사길 방문 전 각 식당 2곳 전화 확인 → 1곳 계단 있음, 1곳 입구 단차 낮아 가능
- 접근 가능한 식당에서 비빔밥 한정식 → 만족도 높음
- 재호 평가: "맛집 지역 찾는 건 이 앱이 최고. 접근성은 어차피 여기서 해결 안 되니 따로 확인"

---

## 평가

| 항목 | 점수 (1-10) | 근거 |
|---|---|---|
| 유용성 | **7** | 현지인 맛집 지역 발굴 기능이 재호의 음식 여행 니즈를 충족. 접근성 정보 부재는 한계이나 재호가 "분리된 문제"로 수용 |
| 사용 편의 | **8** | 개발자 배경으로 앱 탐색 빠름. 데이터 소스 명확성이 신뢰 형성. 코스 생성 UX 직관적 |
| 구현 타당성 | **8** | 현재 API 범위 내 구현 충분히 가능. 접근성 레이어는 별도 API 연동이 필요한 별개 과제 |

**종합 점수: 23/30**

## Pass/Fail 판정

> **✅ PASS**

TasteData Korea는 이재호의 음식 여행 니즈를 충분히 충족한다. 개발자 배경으로 인해 접근성 정보 부재를 "이 앱의 범위 밖"으로 명확히 구분하고 수용하는 성숙한 사용자 반응이 나타났다. 유용성 7점, 종합 23점으로 기준을 상회한다. 장애인 일반 사용자에게는 이 PASS가 적용되지 않을 수 있음을 유의해야 한다.

## 개선 제안

1. 음식 코스 생성 결과에 "이 코스 식당들의 휠체어 접근성 확인되지 않음" 안내 문구 추가 — 장애인 사용자가 현장 방문 전 사전 확인을 놓치지 않도록 명시적 경고
2. 장기적으로 식당 상세 정보에 "입구 단차 없음", "실내 휠체어 이동 가능" 크라우드소싱 태그 도입 — 재호 같은 개발자 사용자가 직접 기여하는 구조로 데이터 품질 향상
