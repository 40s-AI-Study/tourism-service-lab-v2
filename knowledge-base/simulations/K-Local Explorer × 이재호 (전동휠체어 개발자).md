# SIM-57: K-Local Explorer × 이재호 (전동휠체어 개발자)

## 시나리오

이재호(38세, IT 스타트업 백엔드 개발자)는 척수 손상으로 전동휠체어를 사용한다. 여행을 즐기고 SNS에 "휠체어로 가볼 만한 곳" 콘텐츠를 올린다. 팔로워 6,000명의 장애인 여행 커뮤니티 인플루언서. 주말에 경기도·충청도 쪽 숨은 명소를 탐방하려 한다. 앱이 MZ 타겟이라 자신과 감성이 맞을 것 같아 설치해보지만, 무장애 필터가 없다는 것을 곧 발견한다.

---

## 사용자 여정

### 1단계: 진입

앱을 설치하고 관심사 태그를 `#숨은명소`, `#자연경관`, `#포토스팟`, `#주말여행`으로 선택한다. 무장애 관련 태그가 태그 목록에 없다는 것을 즉시 확인한다. `central-attractions-by-municipality` API로 수원·청주 지역 숨은 명소가 추천되지만, 계단이 있는지, 경사로가 있는지, 주차장이 있는지에 대한 정보가 없다.

재호는 개발자 특유의 탐색 방식으로 각 명소 상세 페이지를 꼼꼼히 확인한다. "접근성 정보 없음"이라는 명시적 표시조차 없고, 단순히 해당 필드 자체가 존재하지 않는다.

### 2단계: 행동

대신 사진 공모전 수상작 사진에서 간접적으로 접근성을 판단하려 시도한다. 계단이 없어 보이는 평지 명소, 포장된 길이 보이는 장소를 수동으로 걸러낸다. 시간이 많이 걸리고 불확실하다.

`related-attractions` API가 연계 코스를 제안하지만 이동 동선의 지형(오르막, 비포장) 정보가 없어 코스 전체를 채택하기 어렵다. `visitor-concentration-forecast` API의 혼잡도 정보는 유용하다 — 인파가 많으면 이동이 더 어렵기 때문.

재호는 결국 앱을 참고는 하되 기존에 사용하던 무장애 여행 블로그와 병행해 최종 목적지를 결정한다. "아이디어는 좋은데 나한테는 절반만 쓸모 있다"는 평가.

### 3단계: 결과

방문한 명소 중 한 곳은 사진으로 평지처럼 보였으나 실제로는 입구에 5cm 단차가 있어 전동휠체어 진입이 어려웠다. 재호는 앱 리뷰에 "무장애 필터 추가 요청"을 남기고, SNS에 "좋은 앱인데 장애인 접근성 정보가 없어서 아쉽다"는 글을 올린다. 이 글이 장애인 여행 커뮤니티에서 공유되며 앱에 대한 접근성 개선 요구가 개발팀 이메일로 여러 건 접수된다.

---

## 평가

| 항목 | 점수 (1-10) | 근거 |
|---|---|---|
| 유용성 | **5** | 숨은 명소 발굴 가치 자체는 유용하나 무장애 필터 부재로 절반만 활용 가능. 혼잡도 정보는 이 페르소나에게도 유용. 접근성 정보 부재로 실제 방문 리스크 발생 |
| 사용 편의 | **6** | 개발자 특유의 앱 탐색 능력으로 일반 사용자보다 더 많이 활용. UI 자체는 깔끔하지만 접근성 필드 부재가 핵심 불편 |
| 구현 타당성 | **7** | 무장애 정보 필드 추가는 기술적으로 단순하지만 데이터 수집이 과제. 한국관광공사 무장애 여행 API 또는 크라우드소싱으로 확보 가능성 있음 |

**종합 점수: 18/30**

---

## Pass/Fail 판정

> **✅ PASS** (경계선)

유용성 5점으로 기준(6점) 미달이나, 종합 18점으로 최저선 충족. 그러나 실질적으로 무장애 필터 부재가 이 페르소나의 핵심 사용 장벽이며, 접근성 개선 없이는 장애인 여행자 세그먼트를 의미 있게 서비스할 수 없다. 조건부 PASS로 간주하며 접근성 기능 추가를 강력 권고한다.

---

## 개선 제안

1. **무장애 접근성 필터 및 정보 필드 추가**: 명소 데이터에 계단 여부, 경사로, 장애인 화장실, 전용 주차장, 포장 여부 필드를 추가하고 필터 기능을 제공. 한국관광공사 무장애 여행 데이터 API 연동 또는 사용자 크라우드소싱으로 데이터 구축.
2. **접근성 인증 체계**: 방문한 장애인 여행자가 접근성 정보를 직접 입력·검증하는 커뮤니티 기능 도입. 재호 같은 장애인 여행 인플루언서를 "접근성 앰배서더"로 공식 협력하면 데이터 품질과 커뮤니티 신뢰를 동시에 확보 가능.
