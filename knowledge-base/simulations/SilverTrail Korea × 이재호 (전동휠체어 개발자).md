# SIM-137: SilverTrail Korea × 이재호 (전동휠체어 개발자)

## 시나리오

재호는 장애인 여행 카페 네이버 커뮤니티에서 SilverTrail Korea가 `barrier-free-travel` API를 활용한다는 게시글을 보고 개발자 본능으로 바로 설치했다. 무장애 여행 정보 앱에 항상 목말랐던 그에게 이 앱은 기대와 실망이 교차하는 경험이었다. 시니어 맞춤 무장애 정보는 잘 되어 있었지만, 전동휠체어 특유의 세부 요구사항(경사도 수치, 바닥 재질, 단차 높이, 전동휠체어 충전소)이 누락되어 있었다. "시니어 접근성"과 "전동휠체어 접근성"은 다른 개념인데 그 구분이 없었다.

---

## 사용자 여정

### 1단계: 진입
- 장애인 여행 네이버 카페에서 `barrier-free-travel` API 활용 앱으로 소개 → 개발자 관심 + 실용 기대로 설치
- 앱 실행: 큰 글씨 시니어 UI — 재호에게는 불필요하나 허용. 기능 데이터 품질이 관건

### 2단계: 행동
- `barrier-free-travel` 데이터 탐색: 경주 불국사 무장애 경로 확인 → 경사로 존재 여부는 확인 가능
- 세부 데이터 탐색: 경사도 각도, 바닥 재질(자갈/포장), 단차 높이(mm) — 정보 없음
- 전동휠체어 충전소 검색 → 기능 없음
- `durunubi-trails` 트레일: 시니어 보행 기준 설계 — 전동휠체어 통과 가능 너비(90cm 이상) 정보 없음
- 시니어와 장애인 무장애 기준이 혼용된 데이터 구조 발견 → 개발자 커뮤니티에 "API 데이터 세분화 필요" 이슈 제기

### 3단계: 결과
- 기본적인 무장애 경로 존재 여부 확인에만 부분 활용. 전동휠체어 전용 세부 데이터 부재로 완전 대체 불가. 기존 장애인 카페 후기 병행 사용 지속

## 평가

| 항목 | 점수 (1-10) | 근거 |
|---|---|---|
| 유용성 | **5** | 무장애 경로 존재 여부는 확인 가능. 전동휠체어 특화 세부 데이터(경사도·바닥재질·충전소) 없어 실전 대체 불가 |
| 사용 편의 | **6** | 개발자에게 큰 글씨 UI는 답답하나 허용 범위. 데이터 세분화 부재가 진짜 문제 |
| 구현 타당성 | **8** | `barrier-free-travel` API 기반 무장애 정보 구현은 타당. 장애 유형별 세분화로 개선 여지 명확 |

**종합 점수: 19/30**

## Pass/Fail 판정

> **✅ PASS**

재호는 SilverTrail Korea의 직접 타겟(시니어)은 아니지만 `barrier-free-travel` API 활용 앱으로서 전동휠체어 사용자에게도 부분적 가치를 제공한다. 종합 19점으로 PASS 기준 충족이나 전동휠체어 특화 세부 정보 부재로 완전 전환은 어렵다. 데이터 세분화 개선 시 장애인 여행자라는 새로운 사용자층을 획득할 수 있다.

---

## 개선 제안

1. `barrier-free-travel` 데이터에 장애 유형별 세분화 추가 — "시니어 보행"과 "전동휠체어" 기준을 분리하여 경사도 수치, 바닥 재질, 단차 높이, 통과 너비 별도 표기
2. 전동휠체어 충전소 위치 정보 연동 — 공공 전동휠체어 충전 인프라 데이터 통합으로 장애인 여행자 커뮤니티 전체를 핵심 사용자층으로 확장
